# XMySQL Server P0 上线门槛开发计划

## 1. 目标与范围

本计划用于将当前仓库从“开发/验证可用”推进到“可灰度生产部署”。  
范围仅覆盖 P0 必过门槛：

- 核心高风险文件整改完成
- 崩溃恢复可验证
- 并发正确性可验证
- 基础可观测性上线
- 回滚预案可执行

## 4. 工作流 B：崩溃恢复验证

### 4.1 开发任务

- 建立“异常退出 -> 重启恢复 -> 校验一致性”的自动化脚本
- 覆盖 redo/undo 恢复、半提交事务、回放幂等三类场景
- 对关键表执行前后快照比对与事务结果比对

### 4.2 验收标准

- 异常退出后可恢复到一致状态
- 事务状态正确（已提交保留、未提交回滚）
- 同一恢复场景可重复回放且结果一致

### 4.3 交付物

- 恢复演练脚本
- 演练输入数据集与结果快照
- 崩溃恢复报告

### 4.4 当前状态

- 状态：`已有基础但未达标`
- 现状：
  - 崩溃恢复核心代码和若干测试已存在
- 缺口：
  - 缺自动化恢复脚本
  - 缺 redo/undo/半提交事务正式演练报告
  - 缺一致性校验输出和重复回放证据

## 5. 工作流 C：并发正确性验证

### 5.1 开发任务

- 建立高并发读写压测用例（含冲突写、范围读、长事务）
- 覆盖死锁检测、锁等待、隔离级别关键行为
- 增加并发场景下的数据一致性核对任务

### 5.2 验收标准

- 无死锁风暴与不可恢复阻塞
- 无明显脏读、丢写、重复写
- 压测结束后数据与约束保持一致

### 5.3 交付物

- 并发测试脚本与参数模板
- 一致性核对脚本
- 并发验证报告

### 5.4 当前状态

- 状态：`未实现`
- 现状：
  - 仓库中有零散并发相关测试和 demo
- 缺口：
  - 缺正式并发压测脚本、参数模板和一致性核对脚本
  - 缺冲突写、范围读、长事务场景报告

## 6. 工作流 D：基础可观测性上线

### 6.1 开发任务

- 统一错误日志结构（错误码、模块、请求上下文）
- 补齐慢查询日志（SQL、耗时、影响行数、trace 标识）
- 增加核心指标采集与导出：
  - QPS
  - 延迟分位（P50/P95/P99）
  - 错误率

### 6.2 验收标准

- 错误、慢查询、性能指标均可持续采集
- 可通过日志与指标定位到具体模块与异常路径
- 告警规则可覆盖错误率突增与延迟突增

### 6.3 交付物

- 日志字段规范
- 指标与告警配置
- 可观测性接入说明

### 6.4 当前状态

- 状态：`部分实现`
- 现状：
  - 有部分内部 stats/alert 能力
- 缺口：
  - 缺统一错误日志字段规范
  - 缺慢查询日志可用性证据
  - 缺核心指标导出、告警规则、面板或截图

## 7. 工作流 E：回滚预案与灰度演练

### 7.1 开发任务

- 制定灰度发布策略：只读灰度 -> 低风险写入灰度 -> 扩容
- 制定快速回退流程：版本回退、流量切回、配置回滚
- 制定数据回滚流程：备份恢复点、回放边界、校验步骤
- 开展至少一次全链路演练并记录结果

### 7.2 验收标准

- 灰度步骤可按文档执行且无阻断
- 发生故障时可在预期窗口内完成回退
- 回退后业务可用且数据校验通过

### 7.3 交付物

- 灰度发布手册
- 快速回退手册
- 数据回滚手册
- 演练记录与复盘
